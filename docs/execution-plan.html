Runner · internal working doc

The execution plan, restructured

Zaera Labs' roadmap sketches the app Runner eventually becomes. This is the version scoped to what a solo, session-by-session build can actually do next — reorganized around what needs no new infrastructure versus what needs a real architecture decision first.

“Become the smartest running coach in every runner's pocket.” — that's the North Star, and it means AI isn't optional scope. T2 is confirmed, not a maybe — the only open question is timing.

Where Runner actually stands today

Everything so far is a client-only PWA — no backend, no accounts, no AI calls. T0 and T1 — the entire no-backend build — shipped 2026-07-11. Every item in both tracks below is done.

Shipped: onboarding, plan-eligibility & safety rules, the plan generator (5K–100 mile), editable calendar, cross-training picker, a static workout glossary, PWA install, Today card, workout detail view, structured logging (time/distance/RPE/notes), a Progress/recap panel, a real Settings screen, mi/km units, JSON export/import, illness/vacation mode, difficulty-based volume adaptation, structured pain reporting, and a guided goal-change flow.

Against the roadmap's own "Must Have for Launch" list, everything client-side is now covered. What's left needs a real infrastructure decision — T2 (AI) and T3 (accounts), see below.

T0

Close the Phase 1 gap

The most visible distance between what exists and what the roadmap calls "launch-ready" — all pure UI and logic on top of the generator that already runs.

  • Today screenshipped A card at the top of the calendar shows today's workout and logged status at a glance, instead of relying on scroll-to-highlighted-row.
  • Workout detail viewshipped Tapping any day's date opens what/why/how-hard/if-I-can't/common-mistake, keyed off the generator's structural day type so an edited label can't break which explanation shows.
  • Real logging fieldsshipped Time, distance, RPE 1–10, and notes — the quick calendar time input and the full detail form read/write the same structured entry without clobbering each other's fields. Heart rate skipped for now (needs a device to be worth asking for).
  • Weekly recapshipped A Progress panel: distance/longest-run/sessions so far, last week's planned-vs-completed and hardest session, and a next-week preview. Reuses the same "week before current" logic the adaptive layer already had, so the two never disagree.
  • Settings screenshipped Separate from the plan-edit wizard, which now has its own pencil icon — name, units, export/import, reset plan, delete all data.
  • Export / importshipped JSON download/upload, living in Settings — the intro screen's "everything stays on your device" promise now has an actual backup path behind it.
  • Units toggle (km / mi)shipped Generator still computes everything in miles internally (EVENT_TABLE etc. untouched); a toUnit()/fromUnit() pair converts only at the input/display boundary, so switching units never needed a second copy of the tables.
T1

Deepen adaptation

Still pure client-side logic — no backend — but needs T0's richer logging as an input.

  • Illness & vacation modeshipped Mark date ranges unavailable in Settings; those days become "Rest — illness/away" automatically and are correctly excluded from the missed-workout check (they're typed 'rest', not skipped).
  • Difficulty-based adaptationshipped Consistently too-easy or too-hard RPE on easy/long runs over the last two completed weeks nudges upcoming volume ±5–10%. Shares the "one adaptive story per render" rule with missed-workout dampening — never both at once.
  • Structured pain reportingshipped Location, severity 1–10, worsens-while-running, can-walk-normally — routes to one of three non-diagnostic messages (continue cautiously / back off / get it checked), living in the workout detail view. Never diagnoses.
  • Guided goal-change flowshipped Changing the event, race date, or start date now shows what changes / what stays / new plan length / safety warning before committing — edits that don't touch the race itself still save immediately, no extra step.
T2

The AI fork

Confirmed, not optional — the North Star is being the smartest coach in the runner's pocket, and that's an AI claim. The only open question was ever timing: this starts after T0/T1, once the coaching core is solid enough to be worth explaining. First move is still a small serverless function that proxies LLM calls, so no key ever sits in client code.

  • Stand up the proxy functionshipped Netlify Functions, calling OpenAI (gpt-4o-mini) server-side — the key lives only in Netlify's env, never in client code.
  • "Why this workout" explanationsshipped Grounded only in the active plan's real data (event/goal/phase/week/day type/label/distance) — the AI explains and personalizes, it never invents training load.
  • AI weekly recapshipped A natural-language layer over the T0 recap data, not a replacement for it — the deterministic Progress panel numbers keep working if the AI call fails.
  • Guardrails as a system promptshipped The roadmap's non-negotiable rules (no unsafe plans, no diagnosis, no guilt-based motivation) are literal constraints in every AI function's system prompt, not aspirational copy.
  • Conversational day-swapshipped "I want to run today instead of tomorrow" replaces the old two-day manual retype. The AI only identifies which two already-scheduled days to swap (hard-validated server-side against the real day list); the label swap itself is done client-side with the plan's own data, plus a confirm step before anything saves.
T3

The accounts fork

Also confirmed, also deferred — a bigger backend lift than T2, since it's real auth and a database, not just a proxy function. Comes after T2 proves the AI layer is worth having an account for.

  • Update the intro screen's promiseshipped Now reads "No account required, no ads — your data stays on this device unless you turn on sync," accurate the moment sign-in ships since it's opt-in, not required.
  • Auth + cross-device syncshipped Supabase project `runner` (ref ssqdkituquloolgtlfpl), magic-link email, fully optional — local-only usage is untouched for anyone who never signs in. Sync uses the same lastModified-wins-for-scalars / union-merge-for-collections pattern as Conqur's CloudSync. Real URL + anon key are wired into app.js's SUPABASE_URL/SUPABASE_KEY; first sign-in tested successfully against the live project.
  • Data-deletion flowcode done, needs Netlify env vars Two levels in Settings: "Delete my cloud data" (self-service, deletes the user_data row via RLS) and "Delete my account permanently" (full auth.users deletion via a new Netlify function, netlify/functions/delete-account.js, using a service-role key that never touches the client). Needs SUPABASE_URL, SUPABASE_ANON_KEY, and SUPABASE_SERVICE_ROLE_KEY set as Netlify env vars (same pattern as OPENAI_API_KEY) before this can be tested end to end.
  • Custom SMTP + branded auth emailsdeferred, not urgent Supabase's built-in mailer only sends default-wording templates (confirmed 2026-07-13: the Subject/Body fields in Authentication → Emails are read-only until custom SMTP is connected) — the actual sign-in link inside those emails works fine regardless, this is cosmetic only. Deferred until closer to public launch, since Supabase's built-in mailer is meant for testing/low volume anyway and real users will need reliable delivery at the same time the wording starts mattering. Plan: connect a free-tier SMTP provider (Resend is the easiest Supabase pairing, ~100/day free) then customize the "Confirm signup" and "Magic Link" templates to explain this is Runner's optional sync, not a generic account signup.
T4

Integrations

Realistically depends on T3 — Health Connect, Strava, and Garmin all involve OAuth tokens that need somewhere durable to live.

  • Strava / Garmin / Health Connect import Not a pure-static-PWA task — needs the T3 backend to exist first, regardless of which integration comes first.
  • Workout auto-matching Matching imported activities to planned workouts, with the user able to correct every match — per the roadmap's own rule.
T5

Native app, GPS, notifications

The roadmap itself sequences this last — "reduce dependence on third-party apps only after the coaching product is proven." Nothing changes that logic here.

  • Web Push notifications Lighter first step than a native app — still needs a backend to hold push subscriptions, but far smaller than the rest of this track.
  • Native GPS, live coaching, voice coaching, watch support A real platform decision (native app or a wrapper like Capacitor) — not worth resolving before T0–T2 prove the coaching product itself is worth using daily.

Explicitly not building

Straight from the roadmap's own "do not build without demand" list, and nothing here changes that judgment: community, teams, social feed, public profiles, coach marketplace, leaderboards, virtual races, clubs. If real demand shows up later, that's a new conversation — not a default to design around now.

The actual sequence

Unlike the tracks above, this part is genuinely ordered — each step assumes the ones before it are done.

  1. T0 & T1, all eleven items — done (2026-07-11) Today card, workout detail view, structured logging, Progress/recap panel, Settings screen, export/import, mi/km units, illness/vacation mode, difficulty-based adaptation, structured pain reporting, guided goal-change flow. The entire no-backend build is closed.
  2. T2, all four items — done (2026-07-12) Proxy function, "why this workout" explanations, AI weekly recap, guardrails-as-system-prompt, plus a bonus conversational day-swap. The backend (Netlify Functions + OpenAI) now exists.
  3. T3 — next up The bigger lift: real auth + a database, not just a proxy. Starts with rewriting the intro screen's "no account" promise before it becomes false, per the flag below.
  4. T4, then T5 Genuinely later. Importing Strava data into an app with one user isn't useful yet, and native GPS is something the roadmap itself defers.

Decided

T2 and T3 both happen — the North Star requires it. Timing is the lever, not whether: T0 and T1 come first so the coaching core is solid before there's anything for the AI to explain or an account to protect.

The one piece of homework that comes with that: the intro screen's "no account, no ads, everything stays on your device" line has an expiration date. It's accurate through T0/T1 and stops being true the moment T3 ships — noted as a first step inside T3 above, not left as a surprise for later.
