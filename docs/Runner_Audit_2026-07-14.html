A Executive assessment

Solid coaching core, missing release paperwork

Runner's adaptive-coaching engine — the part of the spec that actually matters — is built with real rigor: the AI never invents a number, every model output is validated server-side before it touches a plan, and a genuine near-miss bug (the coach once defaulting a casual "I want to run" into a hard interval session) was caught and fixed rather than shipped. Closed beta · not production-ready

What's missing isn't architecture or safety logic — it's the surrounding infrastructure a real release needs: legal/medical disclaimer language, a notification layer, and an automated test suite that can catch a future regression the way this session's manual testing caught the last one.

Where it's strongest

  • Never-invent-numbers is enforced, not just prompted. coach.js clamps reduce_intensity to 0.5–0.9, restricts it to easy/long days, and rejects any action referencing a day key that doesn't exist — server-side, regardless of what the model returns.
  • Red flags physically cannot carry a workout. riskLevel:"red" or seek_medical_evaluation hard-nulls the action field before it ever reaches the client.
  • Missed-workout logic matches the spec's safeguards almost line for line — >60% missed dampens future volume ~15%, a missed long run shortens (never doubles) the next one.
  • Illness/vacation mode converts affected days to rest automatically and they're excluded from the missed-workout ratio for free.

Where it falls short

  • No medical/legal disclaimer text exists anywhere — not on the intro screen, not in the Safety panel, not in privacy.html. Section 29 is a flat requirement and it's currently a zero.
  • Zero automated tests. Every verification claim in project history was a manual, one-session spot-check against live OpenAI output — nothing stops a future prompt edit from silently reintroducing the exact bug that was already caught once.
  • No notification layer at all — today's-workout reminders, missed-workout check-ins, race countdowns: none of it exists yet.
  • A collected field (recentRaceTime) is dead code — onboarding asks for it, nothing ever reads it, pace guidance stays RPE-only even when real pace data would be available.

C Architecture assessment

Simple, and correctly separated where it counts

Vanilla JS, no build step, no framework — app.js (2,462 lines) plus seven small Netlify Functions. This is a deliberate choice per §25 ("don't introduce a framework merely because it is popular"), and it's held up: the required three-way system separation is genuinely present in the code, not just described in comments.

  • Deterministic plan engine — EVENT_TABLE, buildStructuredWeeks, computeWeeklyVolumes own every number: phase length, volume progression, cutback weeks, taper curves, safety scaling for compressed timelines.
  • AI interpretation layer — coach.js's system prompt encodes the full red/yellow/green triage, missed-workout rules, and taper/illness/fatigue logic, but only ever outputs a category + optional scale factor.
  • Validation layer — every AI response is re-checked against real day keys, real types, and a clamped factor range before it's allowed to touch the plan; red-flag responses are hard-blocked from carrying an action regardless of model output.

Data lives in localStorage by default; Supabase sync is fully optional and merges with a sensible last-modified-wins-for-scalars / union-for-collections strategy. OAuth tokens are deliberately kept out of the synced blob. Account deletion goes through a service-role Netlify function since it needs privileges the client can't hold. This is good privacy hygiene and matches §28 closely.

Technical debt worth naming: a single 2,462-line file is still "simple" by the letter of §25, but findability is starting to cost real time — several of this audit's gaps (the dead recentRaceTime field, the missing Today-screen affordances) are the kind of thing that hides easily in a file this size. Not urgent, but worth a section-boundary pass (e.g. splitting the plan engine, the AI/coach UI, and cloud sync into separate files) before the codebase grows further.

D Implementation roadmap

  • Add §29's legal/safety disclaimer language to the intro screen, Safety panel, and privacy.html — this is the one gap that should block any wider release on its own.
  • Either wire recentRaceTime into real pace guidance, or remove the onboarding field entirely — a collected-but-unused field is exactly the "placeholder logic called complete" §36 warns against.
MVP completion
  • Add a "preferred long-run day" onboarding input.
  • Turn the handful of scenarios already manually verified this session (red flags, fatigue downgrade, extra-motivation decline, missed-week rebuild, malformed-JSON handling, invented-distance rejection) into a real fixture-based regression suite for coach.js.
  • Give strength/mobility sessions real structured content instead of a generic label.
  • Add baseline nutrition-education content per §21.

  • Rule-based notifications: today's-workout reminder, missed-workout check-in, and race countdown at minimum.

Post-launch

Verdict
Closed beta

Not production-ready — the missing legal/safety disclaimer language is a real blocker on its own, independent of everything else. Not limited-internal-only either: the adaptive-coaching loop, the safety-triage logic, and the data-deletion flows are genuinely solid and already exceed what most first passes at this pattern get right.

Brutally honest: the gaps that remain are legal-text, notification-infrastructure, and content-depth gaps — not architecture or safety-logic gaps. That's the right kind of gap to have at this stage.
